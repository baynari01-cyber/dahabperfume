# DAHAB PERFUMES - المرجعية الهندسية الأساسية للنظام (Core Engineering Specifications)

هذا المستند يمثل المرجعية الهندسية الكاملة لتطوير وبناء نظام "عطور دهب" (Dahab Perfumes). يغطي المعمارية والطبقات، حدود الأنظمة، الهياكل والبيانات وقواعد التحقق للمستودعات والمخازن وإدارة المبيعات والطلبات والاستيراد.

---

## 1. المعمارية العامة (General Architecture)
النمط الأنسب للمشروع هو **Modular Monolith** (تطبيق واحد منظم إلى Modules مستقلة منطقياً).

### هيكل DAHAB System
```
DAHAB System
├── Storefront
├── Admin
├── POS
├── Authentication & Authorization
├── Product Catalog
├── Pricing
├── Finished Product Inventory
├── Raw Material Inventory
├── Perfume Formulas
├── Sales
├── Orders
├── Import System
├── Reports
├── Settings
├── Audit & Security
├── Caching
└── Integrations
```

### طبقات النظام (Architecture Layers)
1. **Presentation Layer**:
   - Storefront UI
   - Admin UI
   - POS UI
2. **Application Layer**:
   - Product Service, Inventory Service, Formula Service, Sales Service, Order Service, Import Service, Settings Service, Report Service.
3. **Domain Layer**:
   - Products, Variants, Formulas, Raw Materials, Sales, Orders, Employees, Permissions.
4. **Infrastructure Layer**:
   - PostgreSQL / Supabase, Prisma ORM, Upstash Redis (Caching), Image Storage, Logging, External Integrations.

## 2. حدود الأنظمة (System Boundaries)

### Storefront (واجهة الزبائن فقط)
* **المسارات**: `/`, `/shop`, `/products/[slug]`, `/collections`, `/cart`, `/checkout`, `/wishlist`, `/about`, `/location`, `/contact`, `/faq`, `/shipping`, `/returns`, `/privacy`, `/terms`, `/order-success`
* **صلاحيات الزبون**:
  - تصفح المنتجات والبحث والتصفية.
  - اختيار الحجم وإضافته للسلة أو قائمة الأمنيات.
  - إرسال الطلب وفتح رسالة واتساب منسقة للطلب.
  - رؤية حالة التوفر العام فقط (متوفر، غير متوفر، متاح عند الطلب).
* **معلومات مخفية عن الزبون**: المواد الخام، التكلفة، حركات المخزون، تقارير المبيعات، حسابات الموظفين، بيانات الصيغ والتركيبات.

### Admin (الإدارة)
* **المسار**: `/admin/*`
* **المسؤوليات**: إدارة المنتجات، التصنيفات، البصمات العطرية، الأسعار العامة، المخزون النهائي والمواد الخام، صيغ التركيب وعمليات الاستيراد، تقارير الأداء والمبيعات، وجرد الخزينة المالي.

### POS / Staff (الموظفون والكاشير)
* **المسار**: `/pos/*` (مثل `/pos/cashier`)
* **قاعدة**: الموظف العادي لا يدخل صفحة `/admin`. إذا كان له صلاحيات للمخزون أو التقارير تظهر له نسخة مخصصة داخل مسار `/pos` بنفس الخدمات والمنطق.

## 3. الوحدات الأساسية والقواعد (Core Modules & Rules)

### 3.1 تسعير المنتجات (Pricing)
* **المصدر الرئيسي**: `GlobalPricingSettings` (تحديد أسعار موحدة للأحجام: 50 مل، 100 مل، 200 مل).
* **قواعد التسعير**:
  - يمنع تسجيل سعر بقيمة صفر.
  - تحفظ الفاتورة نسخة Snapshot من السعر وقت البيع؛ تعديل الأسعار مستقبلاً لا يغير الفواتير التاريخية.

### 3.2 أوضاع المخزون (Inventory Modes)
النظام يدعم وضعي بيع ومخزون:
1. **FINISHED_PRODUCT (جاهز)**:
   - عطر معبأ مسبقاً أو منتج تجاري.
   - عند البيع: خصم كمية `ProductVariant` مباشرة وتسجيل حركة مخزون (`InventoryMovement`).
2. **FORMULA_BASED (تركيب عند الطلب)**:
   - عطر يتم تركيبه يدوياً.
   - عند البيع: جلب صيغة التركيب الفعالة (`ProductFormula`) للمنتج والحجم المختار، ضرب نسب المكونات بالكمية المطلوبة، وخصم المواد الخام المكونة مباشرة من مخزون المواد الخام وتوليد حركة (`RawMaterialMovement`).

### 3.3 صيغ التركيب (Formula Management)
* لكل حجم عطر صيغة تركيب مستقلة تفصيلية (مثال: حجم 50 مل له صيغة مستقلة، ولا يتم ضرب صيغة الـ50 مل تلقائياً لإنتاج الـ100 مل).
* تحتوي الصيغة على: المنتج، الحجم، رقم الإصدار، حالة الفعالية، الحجم الإجمالي، وقائمة المكونات والكميات.

### 3.4 مخزن المواد الخام (Raw Material Inventory)
* **المواد الخام**: زيت عطري، كحول، مثبتات، زجاجات، أغطية، ملصقات، عبوات تغليف وغيرها.
* **قاعدة السوائل**: تحفظ السوائل داخلياً بالملي لتر (ml). تحويل أي إدخال باللتر إلى ملي لتر (1 لتر = 1000 مل) قبل الحفظ.

## 4. هياكل الجداول وقاعدة البيانات (Prisma/DB Schema Reference)
* **Product**: المعرف، SKU، الاسم (عربي/إنجليزي)، التصنيف، الموسم، الجنس، العائلة العطرية، وضع المخزون، الصور، الفعالية.
* **ProductVariant**: المعرف، معرف المنتج، الحجم (50, 100, 200)، السعر الفعلي، كمية المخزون الجاهز.
* **RawMaterial**: المعرف، SKU، الاسم، النوع، الوحدة، الكمية الحالية، حد التنبيه، التكلفة.
* **ProductFormula & ProductFormulaItem**: الصيغة وإصدارها ومكوناتها التفصيلية ونسبة كل مادة خام.
* **InventoryMovement & RawMaterialMovement**: سجل حركات مخازن المنتجات الجاهزة والمواد الخام بالتفصيل (قبل، التعديل، بعد، السبب، والمشغل).
* **Sale & SaleItem**: تفاصيل المبيعات والفواتير مع حفظ نسخ Snapshot من أسماء المنتجات، الأحجام، الأسعار، وضع المخزون، ومعلومات البائع.
* **Order & OrderItem**: سجل طلبات العملاء من المتجر الإلكتروني.
* **Employee & EmployeePermission**: حسابات الموظفين وصلاحياتهم التفصيلية المرفقة على مستوى الخادم (Server-side guards).
* **SiteSettings**: إعدادات النظام بمفتاح وقيمة.
* **AuditLog**: سجل تدقيق أمني للعمليات الحساسة (المستخدم، العمل، البيانات قبل/بعد، الـ IP والمتصفح).

## 5. سير العمليات الحساسة (Critical Workflows)

### 5.1 إتمام مبيعات المنتجات الجاهزة (FINISHED_PRODUCT Sale)
1. قراءة `ProductVariant` مباشرة من قاعدة البيانات.
2. التحقق من توفر المخزون الكافي.
3. بدء معاملة قاعدة بيانات تفصيلية (Database Transaction).
4. إنشاء سجل `Sale` و `SaleItems`.
5. خصم كمية `ProductVariant` وتوليد `InventoryMovement` و `AuditLog`.
6. إتمام المعاملة (Commit). أي فشل يعيد الحالة كاملة (Rollback).

### 5.2 إتمام مبيعات المنتجات المركبة (FORMULA_BASED Sale)
1. جلب الصيغة الفعالة للحجم المطلوب للمنتج.
2. حساب مجموع متطلبات المواد الخام المطلوبة لجميع عناصر السلة.
3. بدء معاملة قاعدة بيانات (Database Transaction).
4. التحقق من توفر كميات المواد الخام المطلوبة في المخزن؛ إذا نقصت مادة واحدة يتم **رفض عملية البيع كاملة**.
5. خصم كميات المواد الخام وتوليد `RawMaterialMovement` لكل مادة و `AuditLog`.
6. إتمام المعاملة (Commit). يمنع الخصم الجزئي أو البيع بدون صيغة مفعلة.

### 5.3 حساب القدرة الإنتاجية (Possible Production Units)
يتم حساب عدد العبوات الممكن إنتاجها لعطر مركب بناءً على كميات المواد الخام المتاحة:
$$\text{Possible Units} = \min \left( \frac{\text{Available Raw Material Quantity}}{\text{Required Quantity Per Unit}} \right)$$
يظهر هذا المؤشر في لوحة الإدارة ونظام الـ POS لمساعدة الكاشير.

## 6. قواعد نظام الاستيراد (Import System Guidelines)
* **المسار**: `/admin/imports`
* **المراحل**: الرفع -> التحليل -> التحقق (Validation) -> المعاينة و Dry Run -> تأكيد الاستيراد الفعلي (Batches Transactions) -> التحقق اللاحق وتحديث الكاش.
* **Dry Run**: إلزامي لعرض تفاصيل التغييرات (الجديد، التحديث، الأخطاء، الصور المفقودة) قبل كتابة أي سطر في قاعدة البيانات.
* **قاعدة التحديث**: إجراء عمليات الاستيراد باستخدام الـ `Upsert` بالاعتماد على الـ `SKU` كمعرف وحيد.
* **تخريب البيانات**: يمنع منعاً باتاً مسح المخزون أو حذف المنتجات الحالية غير المدرجة في ملف الاستيراد.

---

## 7. صلاحيات النظام وأمن الجلسات (Security & Sessions)
* **التحقق من الصلاحيات**: يتم التحقق من الصلاحية (مثل `pos:access`, `settings.pos`, `inventory.counts.review`) على مستوى الخادم (Server-side) في كل Server Action و API.
* **شاشة خمول الكاشير (POS Idle Screen)**:
  - تفعيل شاشة التوقف عند عدم وجود حركة للمدة المحددة.
  - لا تفقد سلة المبيعات المعروضة.
  - اللمسة الأولى تقوم بإغلاق الشاشة فقط دون تفعيل أي زر خلفها.
  - إذا كان رمز الـ PIN مفعلاً، يجب التحقق منه من خلال الخادم.

---

## 8. إدارة الكاش (Caching Strategy)
* تستخدم ذاكرة الكاش (Upstash Redis / Memory cache) **للقراءات العامة فقط** (صفحات المنتجات العامة، التصنيفات، الإعدادات العامة).
* **ممنوع استخدام الكاش في**:
  - عمليات البيع النهائي وتأكيد المعاملات.
  - فحص وتحديث مخزون المنتجات الجاهزة أو المواد الخام.
  - المصادقة والبيانات الأمنية.
* يتم إبطال الكاش (Invalidate) أو زيادة رقم إصدار الكاش (`CACHE_VERSION`) فوراً بعد أي عمليات تعديل منتجات أو مبيعات أو استيراد.
